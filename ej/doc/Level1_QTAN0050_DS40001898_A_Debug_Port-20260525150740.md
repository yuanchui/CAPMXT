

## Using the maXTouch Debug Port

### 1.0 INTRODUCTION

This application note describes the debug port on the maXTouch touchscreen controllers. These devices conform to the object protocol defined in the Protocol Guide for your maXTouch device. The Protocol Guide should be read in conjunction with this application note.

### 2.0 THE DEBUG PORT

### 2.1 Introduction

When the debug port is enabled, a one-directional flow of low level data is sent to the host. Most of this debug data is also available via the normal communications interface, but at a much lower data rate, thus limiting real time observation of signals.

NOTE It can take a considerable amount of time to transmit the debug data over the debug port, so expect the device to operate substantially slower in this mode. Delays of tens of milliseconds can be expected in every cycle. This is normal.

The debug data transmitted by the debug port is typically used during the development of a touch-sensing application. It provides low-level information that is of use in tuning and refining the touch-sensing parameters and the system layout.

### 2.2 SPI Debug Port

The SPI debug port operates over the SPI interface (see Figure 1). The maXTouch device acts as an SPI master that generates data for analysis by an SPI slave. The slave is typically the host chip in a system in which the maXTouch device is a peripheral.

FIGURE 1: SPI DEBUG PORT

<img src="https://cdn.noedgeai.com/bo_d82amsalb0pc73cnab8g_0.jpg?x=638&y=1394&w=638&h=314&r=0"/>

The SPI debug port comprises two main pins: Debug Data and Debug Clock plus, on devices that support it, a Debug Slave Select pin. Refer to the datasheet for your device for the location of the pins.

NOTE On some maXTouch devices, the pins used as the debug port are reassigned GPIO pins. Where this is the case, these pins cannot be used as GPIO pins.

### 2.3 USB Debug Port

Some maXTouch devices use the USB interface to provide debug monitoring via a USB Generic HID. A command is sent to the device and debug-monitoring data packets are returned using the debug port via the USB interface. Note that the DEBUGCTRL and DEBUGCTRL2 fields in the Command Processor T6 must still be used to select which data will be sent (see Section 2.4 "Generating Debug Data").

### 2.4 Generating Debug Data

The maXTouch devices can be commanded to generate debug data via the Command Processor T6 configuration object (refer to the Protocol Guide for your device for more information on this object).

Bytes 4 and 6 of the Command Processor T6 object are the debug controls (DEBUGCTRL and DEBUGCTRL2). By writing to these fields, the host chip can control the generation of debug data for the chip. These fields have the format shown in Table 1.

DEBUGCTRL Field

SIGNAL: Enables the sending of each node signal value.

REFS: Enables the sending of each node reference value.

DELTAS: Enables the sending of each node delta value.

DELTAS8: Enables the sending of each node delta value in 8-bit format.

TSCRN: Enables the sending of the touchscreen data (for example, XY positions, states, and so on) for each supported touch.

CHIP: Enables the sending of the Command Processor T6 general status byte. Refer to your Protocol Guide for details of the status byte in the Command Processor T6 message data.

## DEBUGCTRL2 Field

NOTE This field is available only on T2 and U series devices.

DBGOBJMODEEN: (Debug Object Mode Enable) This is a general control that applies to all objects that support Object Specific Debug Output. When Debug Object Mode is enabled, any of the objects that support Object Specific Debug Output can be enabled to output their own debug data. The format of the debug output will be specific for those objects. If multiple objects are enabled, the order of the data is product specific.

#### 2.5.1 TIMESTAMPS

Generated debug data is always preceded by, and followed by, a one-byte timestamp value. This is a data integrity check for the SPI slave. The start and end timestamps should have the same values.

#### 2.5.2 CHIP STATUS

The chip status information is a one-byte field containing the STATUS field from the Command Processor T6 message data (refer to your Protocol Guide).

#### 2.5.3 TOUCHSCREEN STATUS

The touchscreen status data comprises multiple fields, one per touch, as defined in Table 3.

#### 2.5.4 8-BIT DELTAS

There is an 8-bit delta reported for each node. Each 8-bit delta consists of a signed value that represents the least significant byte of the 16-bit delta for that node.

Values larger than 127 are reported as 127; values smaller than -128 are reported as -128.

QTAN0500

2.5.5 DELTAS

There is a signed 16-bit delta reported for each node.

#### 2.5.6 REFERENCES

There is an unsigned 16-bit reference value reported for each node.

#### 2.5.7 SIGNALS

There is an unsigned 16-bit signal value reported for each node.

#### 2.5.8 DEBUG DATA FORMAT

Debug data bytes are transmitted most significant bit first. 16-bit values are sent least significant byte first.
